# Lumina · 流光 - 塔罗牌网站策划案

## 核心定位

**去掉AI API调用**是一个非常精明的商业决策。这意味着：**零边际成本（Zero API Cost）、极快的响应速度、以及100%可控的内容质量。**

要在这条赛道打败竞品，我们不能拼"理解能力"（因为没有LLM），我们要拼**"呈现的艺术"**和**"心理诱导机制"**。我们要用代码构建一个精密的**"伪随机宿命系统"**。

以下是基于**「纯算法+预设内容库」**架构的完整策划案。

---

## 1. 产品核心概念：「Lumina · 流光」

**定位：** 并不是"算命"，而是**"潜意识投射的数字艺术装置"**。

**核心逻辑：** 利用精密的交互作为"随机数种子"，配合高水准的文学化预设文案，通过巴纳姆效应（Barnum Effect）让用户产生"准到离谱"的错觉。

---

## 2. 机制设计：像游戏引擎一样的塔罗系统

既然没有AI生成，我们需要建立一个结构化的**内容矩阵（Content Matrix）**。

### A. 核心算法逻辑 (The Engine)

我们不使用简单的 `Math.random()`，那太廉价了。我们要创造"命定感"。

#### 1. 混沌洗牌算法 (Chaos Shuffling)

- **Web端：** 追踪鼠标在屏幕上画圈的轨迹、速度、停顿时间。
  - **权限状态：** 100%可行，无需任何权限授权。只要鼠标在网页区域内，浏览器默认允许获取坐标。
  - **实现难度：** 极低，使用标准鼠标事件即可。
  - **体验：** 用户移动鼠标时，实时计算移动速度和距离，累加到随机种子中，形成无法预测的数值。

- **手机端：** 监听 `DeviceMotion` (陀螺仪) 的摇晃力度和角度。
  - **权限状态：** 存在平台差异。
    - **Android：** 大部分浏览器在 HTTPS 环境下默认允许访问，无需弹窗，体验流畅。
    - **iOS 13+：** 需要用户手动授权，必须在用户点击按钮后触发权限请求，会出现系统级弹窗。
  - **解决方案：** 用"仪式感"包装权限请求（详见下方交互设计）。
  - **技术前提：** 网站必须是 HTTPS（安全证书必需）。

- **逻辑：** 将这些物理数据（X, Y, Force, Time）转换成一个 Hash 值，作为抽取卡牌的种子。
- **话术：** "你的每一次抖动，都在重组宇宙的粒子。"（让用户觉得是自己抽出来的，而不是程序随机给的）。

#### 算法实现细节 (The "Fate" Engine)

我们不需要加密级的哈希，我们需要的是**"分布均匀"且"对初值敏感"**的算法。

**核心逻辑：**

1. **数据采样频率：** 每 **100ms** 采样一次（60fps太快，数据量冗余）
2. **收集时长：** 强制 **3000ms (3秒)**
   - *理由：* 少于3秒显得假，多于5秒用户会烦。3秒正好是一个深呼吸的时间
3. **Hash 算法：** 使用 **MurmurHash3** 的简化版或简单的 **FNV-1a** 算法
   - *输入：* `String(X坐标 + Y坐标 + 陀螺仪Alpha + Beta + Gamma + 时间戳)`
   - *输出：* 一个大的整数

**映射逻辑 (Mapping)：**

假设 `TotalCards = 22`

1. 计算最终 Hash 值：`let finalHash = calculateFNV1a(userEntropyString);`
2. 决定哪张牌：`let cardIndex = Math.abs(finalHash) % 22;`
3. 决定正逆位（50/50概率）：检查 Hash 值的最后一位是奇数还是偶数
   - `let isReversed = (finalHash % 2 !== 0);`

#### 2. 标签索引系统 (Tagging System)

- 用户**不能**输入自由文本（因为没有AI去理解）。
- 用户**必须**选择意图标签。
- **分类：** 情感（暗恋/复合/正缘）、事业（跳槽/水逆/财富）、自我（焦虑/指引）。
- **代码逻辑：** `if (tag == 'love') { load_library('love_interpretation'); }`

#### 3. 文案拼装机制 (Text Assembly)

    结果页不是一段死板的话，而是由代码动态拼装的三个模块：

- **模块A（象征）：** 描述牌面视觉（通用）。
- **模块B（核心）：** 对应用户选择的标签（如"复合"）的特定解读。
- **模块C（建议）：** 一个具体的、极易执行的微行动（Action Item）。

### B. 数据结构与内容规划 (Data & Content)

#### MVP版本规划

为了兼顾开发成本与内容质量，**MVP版本建议先做「大阿卡纳」22张**。

**理由：** 大阿卡纳代表人生核心原型，能量最强，解释空间大，且文案工作量可控（22张 vs 78张）。后续迭代再加入小阿卡纳。

#### 卡牌配置

- **数量：** 22张（编号0-21，对应标准大阿卡纳）
- **图片资源：**
  - 每张卡牌 **1张** 高清图（WebP格式，每张30-50KB）
  - **正/逆位处理：** 前端 CSS 控制 `transform: rotate(180deg)`，不需要两张图
  - **背面图：** 全局统一 1 张通用背面（便于建立品牌符号）
    - **设计要求：** 
      - 神秘、优雅的图案（如几何图形、星象符号、或品牌Logo的抽象化）
      - 使用主色调（复古金）和背景色（极暗蓝）的渐变
      - 带有微妙的纹理或光效，支持3D倾斜时的反光效果
      - 尺寸比例与正面卡牌一致（1:1.7）

#### 标签系统 (Tags System)

交互为**单选**（强制用户聚焦问题，减少杂音）。

| 标签ID | 显示名称 | 对应 Context Key | 备注 |
|:---|:---|:---|:---|
| `love_crush` | 暧昧/暗恋 | `context_love` | 侧重不确定性与试探 |
| `love_partner` | 伴侣/复合 | `context_relation` | 侧重沟通与未来走向 |
| `career_change` | 跳槽/迷茫 | `context_career` | 侧重时机与自我价值 |
| `wealth` | 搞钱/财运 | `context_wealth` | 侧重资源与行动力 |
| `self_growth` | 近期指引 | `context_self` | 侧重灵性与潜意识 |

**注意：** 标签ID用于前端交互，Context Key用于JSON数据查找，两者需要一一对应。

所有文案必须预设。逆位（Reversed）必须有独立的解读，因为逆位往往代表"阻滞"或"内化"，文案逻辑完全不同。

## 3. 设计细节与交互流程 (UX & Interaction)

### UI/UX 设计规范

#### 响应式断点

- **Mobile (主战场):** `< 768px` (设计基准：iPhone 13/14, 390px宽)
- **Tablet/Desktop:** `>= 768px` (网页端内容居中，左右留黑，模拟手机比例，或者做成画廊模式)

#### 视觉规范

- **色板：**
  - 背景色：`#050511` (极暗蓝，接近黑)
  - 主色：`#D4AF37` (复古金，用于边框、关键图标)
  - 辅色：`#E6E6FA` (薰衣草雾，用于文字，带85%透明度)

- **字体：**
  - 标题/关键词：`Noto Serif SC` (宋体类，庄重)
  - 正文：`Inter` 或 `PingFang SC` (无衬线，易读)

- **卡牌尺寸：**
  - Mobile: 宽 60vw (屏幕宽度的60%)，比例 1:1.7
  - Desktop: 固定宽 300px

#### 美术资产生产管线 (Visual Production)

风格必须高度统一，不能随便找网图。需要一套**标准化的 AI 绘图指令（Master Prompt）**。

- **风格定义：** 穆夏（Mucha）新艺术风格 + 赛博深空 + 塔罗神秘主义

- **Midjourney V6 Prompt 模版：**

  ```
  [Subject, e.g., A fool walking off a cliff], Tarot card design, Art Nouveau style, Alphonse Mucha mixed with deep space nebula background, glowing gold outlines, midnight blue and obsidian color palette, cinematic lighting, intricate details, mystical atmosphere, 8k resolution, --ar 9:16 --stylize 250 --v 6.0
  ```

- **执行建议：**
  - 用这个Prompt生成全部22张牌，保证色调统一（蓝/紫/金）
  - **牌背设计：** 将 Subject 改为 *"Sacred geometry pattern, eye of providence, galaxy texture"*
  - 所有卡牌图片使用 WebP 格式，每张 30-50KB，确保加载速度

#### 动画规范 (Ritual Motion)

- **洗牌粒子：** 数量 30-50 个微小光点，随手指移动，寿命 1秒
- **翻牌动画：**
  - 总时长：1.2秒
  - 过程：牌先浮起 (0.3s) → 快速翻转 (0.4s, ease-in-out) → 缓慢落下并发光 (0.5s)
- **性能优化：**
  - 所有动画必须使用 `transform` 和 `opacity`，绝对不要改变 `top/left` 或 `width/height`
  - 给正在动画的元素加上 `will-change: transform`

#### 听觉设计 (Sound Design)

视觉只能占领眼睛，**声音才能占领大脑**。大部分竞品是静音的，这是巨大的机会。我们需要构建一个**"声场"**，而不是简单的背景音乐。

- **背景底噪 (Ambience):**
  - **不要：** 吵闹的神秘音乐或廉价的合成器循环
  - **要：** **432Hz 疗愈频率** 混合一点点 **粉红噪音（Pink Noise）** 或 **雨夜声**
  - **作用：** 这种声音能让人迅速平静（Alpha波），降低心理防线，增加对结果的信服度
  - **音量控制：** 背景音保持在 20-30% 音量，不干扰用户思考

- **交互音效 (UI SFX):**
  - **点击标签：** 不是电子的"滴"声，而是**"水滴滴入湖面"**的空灵声
  - **洗牌过程：** 真实的**"厚纸牌摩擦"**的沙沙声（ASMR级别），随手指移动动态变化
  - **翻牌瞬间：** 重低音的**"空气震动"**声（Whoosh）+ 风铃的尾音
  - **技术实现：** 使用 `Howler.js` 管理音频，确保无延迟，支持预加载和淡入淡出
  - **性能考虑：** 音频文件使用压缩格式（OGG/MP3），总大小控制在 500KB 以内

### 阶段一：沉浸式入口 (Hypnotic Entry)

- **视觉：** 黑色背景，中心悬浮一张**全息质感**的卡牌背面。随着手机倾斜，卡牌反光会流动（利用 CSS 3D Transform 或 Three.js）。
- **文案：** "向宇宙提问前，请先诚实面对自己。"
- **交互：** 下方出现一组极简的浮动标签（Tag Cloud）。用户点击标签后，标签发光并吸入卡牌中心（单选交互）。

### 阶段二：物理化洗牌 (Physical Ritual)

#### 权限引导流程（iOS 关键步骤）

由于 iOS 系统需要用户点击按钮后才能请求陀螺仪权限，我们将技术限制包装成魔法仪式，避免生硬的权限弹窗打断体验。

**第一步：引导页（The Cover）**
- 画面上有一个神秘的、发光的指纹按钮或法阵按钮。
- **文案：** "点击注入能量，建立链接。"（实际上这是一个触发权限请求的按钮）。
- **视觉：** 按钮带有呼吸灯效果，暗示需要用户主动激活。

**第二步：点击触发（The Trigger）**
- 用户点击按钮后，代码立即调用权限请求接口。
- **iOS用户：** 此时会看到系统弹窗"是否允许访问动作与方向"。
- **心理策略：** 因为刚才的文案说是"建立链接"，用户会潜意识认为这是链接宇宙的必要步骤，点击"允许"的概率极高。
- **Android用户：** 直接跳过权限弹窗，进入下一步。

**第三步：摇晃洗牌（The Action）**
- 权限获取成功后，进入洗牌页面。
- **提示文案：** "现在，摇晃你的手机，让能量流动。"
- **视觉反馈：** 
  - 实时显示摇晃力度和角度，增强参与感
  - **进度条：** 显示3秒倒计时，使用圆形进度条或线性进度条，颜色为主色（复古金），带有呼吸动画效果
  - **进度提示：** 当进度达到50%时显示"能量正在汇聚..."，达到80%时显示"即将完成..."

#### 核心交互体验

- **手机端核心亮点：** 利用 H5 的 Vibration API。
  - 用户手指按住卡牌堆，开始模拟"切牌"。
  - 手指每滑过一张牌，手机就**短震动**一次（哒-哒-哒）。
  - **震动参数：** 每次震动时长 50ms，震动强度中等（避免过于强烈）
  - **震动模式：** 仅在触摸滑动时触发，摇晃时不震动（避免过度干扰）
  - 这种触觉反馈会极大增加真实感和紧张感。
- **视觉：** 牌堆被打乱，粒子特效跟随手指轨迹或手机摇晃方向。

### 阶段三：揭晓与解读 (The Reveal)

- **翻牌动画：** 慢！一定要慢。先翻开一角，停顿0.5秒，再完全翻开。
- **结果页布局：**
  1. **高清牌面图**（占据屏幕40%）。
  2. **关键词：** 从卡牌的 `keywords` 数组中随机选择1-2个显示（用衬线体大字）。
  3. **拼装文案：** 也就是上面提到的模块A+B+C。
  4. **"护身符"生成按钮：** 引导用户保存图片。

### 结果页与分享功能 (The Viral Asset)

这是最重要的**"产品交付物"**。

#### 结果卡片布局（生成图片规格）

- **尺寸：** **1080 x 1920 px** (完美适配 Instagram/Red Story)
- **格式：** JPG (质量 0.9)，体积控制在 300KB 以内
- **布局结构：**
  1. **Top (15%):** 日期（YYYY.MM.DD） | 月相图标 | "Lumina · 流光" Logo
  2. **Visual (40%):** 巨大的卡牌插画（正/逆位）
  3. **Message (35%):**
     - 大标题：关键词（如"破晓"）
     - 正文：3-4行诗意的冷读文案（模块A+B+C）
     - Action：一行小字建议
  4. **Footer (10%):** 二维码 + "长按测试你的运势"

#### 结果页的"社交货币"挂件 (Social Anchors)

在结果页（卡片）上，除了牌面和文案，还必须增加两个**"毫无逻辑但极具传播力"**的小挂件。这是小红书用户最喜欢互相对比的东西。

- **要素 A：今日灵数 (Angel Number)**
  - **逻辑：** 随机生成一个三位数或四位数（如 111, 333, 777, 888, 1212）
  - **文案：** "宇宙给你的暗号：**888**"
  - **作用：** 评论区会刷屏："我是888接好运！"
  - **实现：** 基于卡牌ID和Hash值生成，确保每次抽牌结果一致
  - **显示位置：** 结果页右上角，小字显示

- **要素 B：能量颜色 (Aura Color)**
  - **逻辑：** 根据牌面随机分配一个颜色代码（如 #FFD700 金色）
  - **展示：** 一个模糊的光晕球体，带有微妙的呼吸动画
  - **文案：** "今日幸运色：**香槟金**"
  - **作用：** 给用户穿搭或购买周边的心理暗示
  - **实现：** 预设22种颜色，对应22张卡牌，或基于Hash值生成
  - **显示位置：** 结果页左上角，与灵数对称

- **设计要点：**
  - 两个挂件都要出现在分享卡片上
  - 使用微妙的动画效果（呼吸、闪烁）增强神秘感
  - 颜色和数字要足够"吉利"，符合用户心理预期

#### 月相计算 (Moon Phase)

- 不需要API，使用简单的算法库 `lunar-phase-js` 或者手写一个简单的近似算法（基于已知的新月日期推算周期29.53天）
- **显示形式：** 仅显示图标（🌑 🌒 🌓 🌔 🌕 🌖 🌗 🌘）

#### 二维码内容

- 链接到网站首页，或包含特定参数（如 `?card=0&reversed=true`）用于追踪分享效果

## 4. 为什么能打败竞品？（差异化）

### 1. 文案的"冷读术"水准 (Cold Reading)

现有竞品（甚至AI）的文案往往太具体或太啰嗦。我们要写出**"诗意且模棱两可"**的文案。

- **坏文案：** "你下个月会遇到一个穿红衣服的人。"（不准会被骂）
- **好文案（我们的）：** "你正在等待一个信号，但其实答案早就写在了你上周犹豫的那瞬间。放下对完美的执念，不完美才是契机。"
- **原理：** 用户会自动把自己的人生代入这段话，觉得"它懂我"。

### 2. 极致的视觉与分享欲 (Shareability)

小红书用户是"视觉动物"。

- **包含元素：** 牌面 + 今天的日期 + 月相 + 一句英文格言 + 二维码。

- **图片生成实现细节：**
  - 使用 `html2canvas` 库将结果卡片DOM元素转换为Canvas
  - 生成前隐藏所有交互元素（按钮、滚动条等），只保留视觉内容
  - 设置Canvas尺寸为 1080 x 1920px，DPI缩放比例 2（保证高清）
  - 转换为JPG格式，质量设置为 0.9，体积控制在 300KB 以内
  - 生成过程中显示"正在生成你的护身符..."提示
  - 生成完成后自动触发下载，或显示预览让用户长按保存

## 6. 技术栈推荐 (Lightweight)

为了实现极佳的动画和跨端体验：

- **框架：** Vue 3 或 React (Next.js)
- **动画库：** Framer Motion (做UI动画) + Three.js (如果想做很炫的3D牌，或者简单的CSS3D即可)
- **打包：** Vite (极速)
- **部署：** Vercel 或 Netlify (免费且全球CDN加速，完全不需要复杂的后端服务器，只需要静态托管)

### 性能与优化

#### 资源加载策略

- **JSON：** `main.json` (包含所有22张牌文案) 体积很小 (<50KB)，**一次性加载**
- **图片：**
  - 首页加载时，**静默预加载**所有22张卡牌图片（WebP格式，每张30-50KB，总计约1MB）
  - **Loading页设计：**
    - 黑色背景，中心显示一个**呼吸光圈**（复古金色，带透明度渐变）
    - 光圈大小从 60px 到 120px 循环变化，周期 2秒，使用 ease-in-out 缓动
    - 光圈中心显示"Lumina · 流光" Logo 或卡牌背面预览
    - 底部显示加载进度百分比（可选）
    - 文案："正在连接宇宙能量..." 或 "准备你的塔罗之旅..."
  - 使用 `loading="lazy"` 或手动控制加载时机

#### "加载中"的心理诱导文案 (The 'Loading' Theater)

既然是纯前端计算，真正的结果在 10ms 内就出来了。但你**绝对不能**马上显示。你需要**人为制造 2-3 秒的等待**，并在这期间展示一套**"伪技术+伪玄学"**的加载文案，让用户觉得系统在进行复杂的运算。

**Loading 序列（每 0.8 秒切换一句）：**

1. `正在链接潜意识频率...` (Connecting frequency...)
2. `捕捉环境磁场熵值...` (Capturing entropy...)
3. `星图矩阵校准中...` (Aligning star matrix...)
4. `解读完成。` (Revelation ready.)

**设计意图：** 这些术语虽然是编的，但听起来很像"量子力学+神秘学"，能极大提升产品的**高智感**。

**技术实现：**
- 使用 `setTimeout` 控制文案切换时间
- 文案淡入淡出动画（opacity 0 → 1，0.3s）
- 最后一句"解读完成"后延迟 0.5 秒再显示结果

#### 动画性能优化

- **GPU加速：** 所有动画必须使用 `transform` 和 `opacity`，绝对不要改变 `top/left` 或 `width/height`，否则会引起回流（Reflow）导致卡顿
- **Will-change：** 给正在动画的元素加上 `will-change: transform`
- **节流：** 鼠标移动和陀螺仪事件需要节流，避免过度计算

### 技术实现注意事项

- **HTTPS 必需：** 陀螺仪权限和震动 API 都要求网站必须是 HTTPS，部署时确保 SSL 证书配置正确。
- **权限降级方案：** 如果用户拒绝权限或设备不支持，可以降级为触摸滑动洗牌，通过手指轨迹生成随机种子，保证所有用户都能完成体验。
- **跨平台兼容：** 需要检测设备类型和浏览器支持情况，针对不同平台提供不同的交互方式（鼠标/陀螺仪/触摸）。

### 错误处理与边界情况

#### 权限拒绝 (iOS)

1. **检测：** `catch` 到 `permission denied`
2. **降级：** 隐藏"摇一摇"提示，显示"请用手指在屏幕画圆"
3. **交互：** 启用 Canvas 监听 `touchmove`，通过手指轨迹生成随机种子

#### 数据不足

- **判断：** 如果3秒到了，但 `eventCount` < 10 (用户没动)
- **反馈：** 进度条卡住，文字提示"能量链接中断，请继续动作"
- **处理：** 延长收集时间，或使用时间戳作为补充熵源

#### 浏览器兼容性

- **目标：** iOS Safari 13+, Chrome 85+, 微信内置浏览器
- **Polyfill：** 如果不支持 ES6+ 语法，需用 Babel 转译。但建议直接放弃过老机型，保证视觉效果

#### 设备不支持

- **陀螺仪不支持：** 降级为触摸滑动洗牌
- **震动不支持：** 静默跳过，不影响核心体验
- **Canvas不支持：** 显示错误提示，引导用户更新浏览器
